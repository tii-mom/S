# SHORE Data Model Baseline

## 1. Status

No production D1 migration is introduced in Phase 0. This document freezes the rules that future migrations must follow.

## 2. Core tables planned for the first operational data phase

```text
users
sessions
debt_summaries
tasks
task_submissions
points_ledger
wallets
referrals
notifications
risk_flags
audit_logs
```

TON integration later adds:

```text
chain_contracts
chain_events
chain_sync_cursors
genesis_purchases
vesting_entitlements
token_claims
rounds
round_metrics
round_approvals
```

## 3. Global rules

- Primary keys are stable text identifiers generated by the application.
- All timestamps use UTC ISO strings or integer Unix milliseconds consistently per table.
- Token and money values use integer smallest units or decimal strings.
- No balance uses SQLite `REAL` or JavaScript floating point.
- User-visible writes include an idempotency key.
- Soft deletion is not a substitute for required private-file deletion.
- Administrative writes create an audit entry in the same logical operation.

## 4. Ledger rules

`points_ledger` and future asset ledgers are append-only.

A ledger entry contains at least:

```text
id
user_id
entry_type
amount
reference_type
reference_id
idempotency_key
created_at
created_by
reason
```

Allowed point entry types begin with:

```text
earn
reverse
adjust
```

The displayed balance is derived from ledger entries. A cached balance may exist for performance, but a reconciliation job must be able to recompute it.

## 5. Debt data boundary

`debt_summaries` stores only the user-confirmed aggregate required by the product flow:

```text
id
user_id
currency
ai_detected_amount_minor
confirmed_amount_minor
item_count
status
confirmed_at
created_at
updated_at
```

Raw screenshots and extracted sensitive text do not belong in this table. R2 object references and analysis jobs are introduced separately with limited retention.

## 6. Task submission state

```text
pending
ai_review
manual_review
approved
rejected
resubmission_required
reversed
```

Transitions must be explicit. Approval and point issuance must be idempotent so retries cannot grant the reward twice.

## 7. Chain synchronization

D1 is not the authority for on-chain balances. The chain event tables are replayable projections.

Each event record includes:

```text
network
contract_address
transaction_hash
logical_time
message_hash
event_type
payload_json
observed_at
confirmed_at
```

The unique identity must prevent a replayed indexer batch from duplicating a purchase, entitlement or claim.

## 8. Migration policy

Every migration must pass:

1. fresh database application;
2. upgrade from the previous committed schema;
3. repeated application safety where applicable;
4. data preservation check;
5. forward-fix or rollback procedure.

Production migrations are never edited after application. Corrections use a new migration.
