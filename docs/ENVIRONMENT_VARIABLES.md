# SHORE Environment Variables

## Web

| Variable                              | Local example                                    | Purpose                          |
| ------------------------------------- | ------------------------------------------------ | -------------------------------- |
| `NEXT_PUBLIC_SHORE_API_BASE_URL`      | `http://localhost:8787`                          | Browser-visible SHORE API origin |
| `NEXT_PUBLIC_TONCONNECT_MANIFEST_URL` | `http://localhost:3000/tonconnect-manifest.json` | Public TON Connect manifest URL  |

The Web runtime defaults to `http://localhost:8787` locally. Staging and Production must set explicit HTTPS origins during build/deployment.

## API vars

| Variable                       | Local example                                 | Purpose                                                         |
| ------------------------------ | --------------------------------------------- | --------------------------------------------------------------- |
| `APP_ENV`                      | `local`                                       | `local`, `staging` or `production`                              |
| `CORS_ORIGIN`                  | `http://localhost:3000,http://127.0.0.1:3100` | Comma-separated allowed Web origins                             |
| `LOG_LEVEL`                    | `debug`                                       | Structured log threshold                                        |
| `TON_APP_DOMAIN`               | `localhost`                                   | Exact domain expected in `ton_proof`                            |
| `TON_NETWORK`                  | `testnet`                                     | `testnet` or `mainnet`; current project policy requires Testnet |
| `SHORE_CLAIM_CONTRACT_ADDRESS` | empty                                         | Testnet claim contract address; empty means fail-closed         |

`SHORE_CLAIM_CONTRACT_ADDRESS` must remain empty until the contract address and message schema are both reviewed and deployed.

## API secrets

The following values must be configured with Cloudflare Secrets and never placed in Wrangler `vars`, source files, screenshots, logs, documentation examples containing real values, or Git history:

```text
TELEGRAM_BOT_TOKEN
ADMIN_REVIEW_TOKEN
```

Set them per environment, for example through Wrangler's secret command or the Cloudflare dashboard. Local development uses dedicated non-production values in `apps/api/.dev.vars`, which is ignored by Git.

## Cloudflare bindings

The API expects:

```text
DB            D1Database
PRIVATE_FILES R2Bucket
ASYNC_JOBS    Queue producer and consumer
```

The Queue configuration also uses a dead-letter queue for exhausted retries.

The Web application expects:

```text
ASSETS        Workers Assets binding generated by OpenNext
```

## Environment policy

### Local

- Staging bootstrap session allowed.
- Local D1/R2/Queue allowed.
- TON Testnet only.
- Dedicated test credentials only.

### Staging

- Telegram authentication enabled with a Staging Bot credential.
- Administrator review enabled with a Staging-only credential.
- TON Testnet only.
- Real mainnet assets forbidden.

### Production

- Staging bootstrap session disabled.
- Telegram `initData` verification required.
- Mainnet claims remain disabled until separate launch approval.
- Secrets must be rotated after suspected exposure.

## Local file

Create `apps/api/.dev.vars` manually when Telegram or administrator flows need to be tested locally. Do not copy production secrets into local development.
